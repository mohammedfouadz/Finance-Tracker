## 📐 Conversion Rules (apply consistently)

### Rule 1 — Import the hook
At the top of every page/component you convert, add:
```tsx
import { useI18n } from "@/lib/i18n";
```

Inside the component:
```tsx
const { t, lang, isRtl, formatDate, formatNumber } = useI18n();
```

### Rule 2 — Replace hardcoded English with `t()`
Find hardcoded strings inside JSX (`>Some Text<`, `placeholder="..."`, `title="..."`, `<CardTitle>...</CardTitle>`, etc.) and replace with the matching translation key from `i18n.tsx`.

**Lookup pattern**: keys are namespaced by page. For Income page → use `t("income.xxx")`. For Dashboard → `t("dashboard.xxx")`. Check `i18n.tsx` for the exact key. If a needed key is missing, **add it to BOTH `en` and `ar` blocks** in `i18n.tsx` and reuse it.

**Example transform**:
```tsx
// BEFORE
<h2>Total Income</h2>
<p>Track your income sources.</p>
<Button>Add Income</Button>

// AFTER
<h2>{t("income.totalIncome")}</h2>
<p>{t("income.subtitle")}</p>
<Button>{t("income.addIncome")}</Button>
```

### Rule 3 — Remove `isAr ? "..." : "..."` patterns
The `dashboard.tsx` and `reports-ai.tsx` files use the old pattern `isAr ? "عربي" : "English"`. Replace ALL of these with `t("...")` calls.

```tsx
// BEFORE
<span>{isAr ? "إجمالي الثروة" : "Total Wealth"}</span>

// AFTER
<span>{t("dashboard.totalWealth")}</span>
```

After converting, you can also delete the local `isAr = lang === "ar"` variable if no longer used.

### Rule 4 — Use logical CSS properties (RTL-safe)
Replace direction-specific Tailwind classes with their **logical** equivalents:

| Old (LTR-only)            | New (RTL-aware) |
|---------------------------|-----------------|
| `ml-X`                    | `ms-X`          |
| `mr-X`                    | `me-X`          |
| `pl-X`                    | `ps-X`          |
| `pr-X`                    | `pe-X`          |
| `left-X`                  | `start-X`       |
| `right-X`                 | `end-X`         |
| `text-left`               | `text-start`    |
| `text-right`              | `text-end`      |
| `border-l`                | `border-s`      |
| `border-r`                | `border-e`      |
| `rounded-l-xl`            | `rounded-s-xl`  |
| `rounded-r-xl`            | `rounded-e-xl`  |

**Don't change** `text-center`, `mx-auto`, `gap-X`, `flex`, `grid` — these are direction-neutral.

### Rule 5 — Format dates with `formatDate`
Anywhere you see `format(date, "MMMM d, yyyy")` from `date-fns`, switch to:

```tsx
{formatDate(t.date, "long")}    // "29 أبريل 2026" or "April 29, 2026"
{formatDate(t.date, "short")}   // "29 أبر 2026"  or  "Apr 29, 2026"
```

For chart axis labels, you can keep `date-fns` `format()` since charts often need short month abbreviations — but consider switching critical UI dates.

### Rule 6 — Keep currency numbers LTR
Numbers, currency amounts, and IBAN/account numbers should always render LTR even inside RTL text. Wrap them with `dir="ltr"` or use the `<span dir="ltr">` pattern. The provided CSS file has helper rules — just add `data-currency` attribute or `currency-amount` className to money displays:

```tsx
<span className="currency-amount font-bold">{formatAmount(value)}</span>
```

### Rule 7 — Form input directions
For email, phone, password, URL, and number inputs, ensure they stay LTR even in Arabic mode. The CSS handles most cases automatically. If you need explicit control, add `dir="ltr"` to the input.

### Rule 8 — Icons that need flipping in RTL
ChevronRight/Left and ArrowRight/Left should flip. The CSS handles this automatically for `lucide-react` icons. If you have an icon that should NOT flip (e.g., a logo), add the className `no-flip`:
```tsx
<ChevronRight className="no-flip w-4 h-4" />
```

### Rule 9 — Sidebar / Sheet sides
The Sheet (mobile drawer) should open from the correct side based on language. Use:
```tsx
<SheetContent side={lang === "ar" ? "right" : "left"}>
```

### Rule 10 — Arabic numerals (optional, contextual)
For typical financial UI, **keep Latin digits** (`1234.56`) — this is the convention even in Arabic financial software because it's clearer. Only use Arabic-Indic digits (`١٢٣٤.٥٦`) for ceremonial/non-numeric contexts.

If you DO want Arabic digits in a specific place:
```tsx
{formatNumber(value, true)}  // ١٬٢٣٤
```

## 🚨 Important Constraints

1. **DO NOT translate**:
   - Backend/database field names (e.g., `categoryId`, `userId`, `firstName` as a property)
   - API endpoints
   - URLs / route paths (e.g., `/dashboard`, `/zakat`)
   - `data-testid` values
   - Console logs / errors meant for developers
   - File names / file contents not user-facing
   - Brand name "Wealthly" (keep it as-is)

2. **DO translate**:
   - All visible text in JSX
   - `placeholder=`, `title=`, `aria-label=`, `alt=` attributes
   - Toast messages (`toast({ title: "..." })`)
   - Error messages shown to users
   - Confirm/alert dialog text
   - Empty-state messages ("No data", "Nothing to show", etc.)
   - Button labels
   - Form labels
   - Chart labels and tooltips

3. **Country names**: I included an Arabic-mapped country list in the new `auth.tsx`. Reuse this approach if other pages have country dropdowns — backend stays receiving English canonical names.

4. **Categories from database**: User-created category names (Food, Transport, etc.) come from the database and shouldn't be force-translated. However, **system-generated default categories** can be detected by name and shown via translation keys. Add this logic only if you can identify them safely.

5. **`document.title`**: Set the page title dynamically based on language:
   ```tsx
   useEffect(() => { document.title = `${t("nav.dashboard")} — Wealthly`; }, [t]);
   ```

## 📝 Code Style

- Always use `t("namespace.key")` — never inline `lang === "ar" ? ... : ...`.
- Always use logical CSS (`ms-X`, `me-X`, `start-X`, `end-X`).
- Always destructure the hook once at the top of the component:
  ```tsx
  const { t, lang, isRtl, formatDate, formatNumber } = useI18n();
  ```
- Never write a translation key in only one language. If you add `"foo.bar": "Hello"` to `en`, you MUST also add `"foo.bar": "مرحباً"` to `ar`.

## 💬 Tone Guide for New Arabic Translations

If you need to add new Arabic strings, follow these conventions:

- **Tone**: Modern Standard Arabic (MSA), neutral and professional
- **Voice**: Address user directly ("أدر أموالك" — manage YOUR money) — use second-person
- **Finance terms**: Use established Arabic finance terminology, not literal translations:
  - "Net Worth" → "صافي الثروة" (not "القيمة الصافية")
  - "Liabilities" → "الالتزامات" (not "المطلوبات" unless very formal context)
  - "Expense" → "مصروف" (singular) / "مصروفات" (plural)
  - "Asset" → "أصل" / "أصول"
  - "Debt" → "دين" / "ديون"
  - "Cashflow" → "التدفق النقدي"
  - "Bank Account" → "حساب بنكي"
  - "Savings" → "ادخار" / "مدخرات"
- **Islamic terms** (use precise religious vocabulary):
  - "Zakat" → "الزكاة"
  - "Nisab" → "النصاب"
  - "Hawl" → "الحول"
  - "Zakatable" → "ما تجب فيه الزكاة" / "زكوي"
- **Punctuation**: Use Arabic comma `،` and Arabic question mark `؟` where appropriate. Use ellipsis `...` (3 dots) for "Loading...".
